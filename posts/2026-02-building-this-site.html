Building This Site — Design Decisions in Plain HTML and CSS

This post documents how and why this site is built the way it is. No framework, no build step, no JavaScript beyond a small activity grid script. The constraint was deliberate.

The site needed to go live fast and stay maintainable without tooling overhead. A curriculum site that requires a Node environment to edit is a site you stop editing. Plain HTML files mean any text editor, any machine, any connection, immediate preview. The complexity budget went into the CSS, not the stack.

The palette: one constraint, carried everywhere

Black background, white type, IBM Plex Mono throughout. No color. The constraint came from wanting the activity grid on the Writing page to do the visual work — a GitHub-style contribution calendar that grades from black to white as commit density increases. Color anywhere else would compete with it. Monochrome everywhere makes the grid legible.

The full palette lives in CSS custom properties at the top of style.css:

:root {
  --black:  #000000;
  --white:  #ffffff;
  --gray-0: #f8f8f8;
  --gray-1: #e8e8e8;  /* --text: body copy */
  --gray-2: #c0c0c0;  /* --text-2: secondary */
  --gray-3: #888888;  /* --text-3: labels, meta */
  --gray-4: #444444;  /* --border-2: active borders */
  --gray-5: #222222;  /* --surface-2, --border */
  --gray-6: #111111;  /* --surface: card backgrounds */

  --bg:     var(--black);
  --hi:     var(--white);  /* headings, links, emphasis */
}

Naming the semantic aliases (--text, --border, --surface) separately from the raw grays means a future light-mode toggle swaps eight values in one place and nothing else changes.

Typography: one font, three weights

IBM Plex Mono in regular (400), medium (500), and italic. The monospace constraint reinforces the technical register without reading as a code dump — Plex Mono has enough optical spacing at small sizes to work as body text, which most monospace fonts don't. The base font size is 15px, slightly larger than most developer sites, because the line lengths here are long and the content is dense.

Layout: two max-widths, no grid framework

Two layout classes handle everything. .page at 800px for reading-focused pages (About, Writing, posts). .page-wide at 1020px for the Curriculum page which needs room for the course card tables. Both center with margin: 0 auto and use a single --pad variable (1.5rem desktop, 1rem mobile) for all horizontal padding.

.page      { max-width: 800px;  margin: 0 auto; padding: 3.5rem var(--pad) 6rem; }
.page-wide { max-width: 1020px; margin: 0 auto; padding: 3.5rem var(--pad) 6rem; }

Course cards: a table inside a bordered div

Each course on the Curriculum page is a .course div containing a header row and a .course-rows table. The header carries the course code, title, and tag badges. The table rows hold the structured metadata (Covers, Books, Note, Pair with). Using an actual table for the rows — rather than flexbox or grid — means the label column is a fixed 80px and the content column fills the remainder, without any JavaScript measurement.

.course-rows td:first-child {
  font-size: 0.65rem;
  letter-spacing: 0.08em;
  text-transform: uppercase;
  color: var(--text-3);
  width: 80px;
  white-space: nowrap;
}

The framework explainer: CSS Grid with auto-fit

The PAC/IPO/GOM/GWM reference block on the Curriculum page uses grid-template-columns: repeat(auto-fit, minmax(220px, 1fr)). This gives two columns on desktop and collapses to one on narrow viewports with no media query — the grid figures it out from the available space. Each cell has a right border and bottom border; the last cell in the last row has neither. Getting border removal right on the last items without JavaScript requires the nth-last-child selector:

.framework-grid {
  display: grid;
  grid-template-columns: repeat(auto-fit, minmax(220px, 1fr));
}

/* remove bottom border from last row (2 items) */
.framework-grid .framework-item:nth-last-child(-n+2) {
  border-bottom: none;
}

/* single-column fallback: remove all right borders */
@media (max-width: 500px) {
  .framework-grid { grid-template-columns: 1fr; }
  .framework-item { border-right: none; }
}

The activity calendar: GitHub SVG, CSS filter

The Writing page originally used a JavaScript-generated grid of div squares with hardcoded activity levels. That was replaced with a live GitHub contribution chart from ghchart.rshah.org, which serves your contribution graph as an SVG image. The image is green by default. Three CSS filter operations convert it to the site palette:

.gh-calendar-img {
  filter: grayscale(1) invert(1) contrast(1.2);
  /* grayscale(1)  — strips green entirely         */
  /* invert(1)     — flips to white-on-black        */
  /* contrast(1.2) — sharpens cell boundaries       */
}

The result is a live calendar that updates automatically with every commit, rendered entirely in CSS, with no JavaScript and no backend.

What comes next

The site is static by choice for now, not by limitation. When the Shopify app reaches production it will make sense to revisit — a Vite build step would enable component reuse across the post template, and a small API call could replace the placeholder post dates with real commit timestamps. Neither is worth adding before there is real content to maintain.

The constraint that actually mattered: every page loads instantly, works without JavaScript enabled, and can be edited in any environment including a WSL2 terminal with no GUI. For a site about systems thinking, that seemed like the right starting point.
