Building This Site

This is the second version of this portfolio site — static HTML, a hand-rolled CSS system, and a deliberate decision to keep it that way until the right reason to migrate appeared. That reason is now here. This post documents what was built, why it was built that way, and what the migration to Next.js / React / TypeScript involves.

February 2026· site-2 → site-3· static HTML · CSS · Next.js · React · TypeScript

Part 1 — The Static Site

The first version of this site was a single HTML file. The second version — this one — expanded to nine files sharing one stylesheet. The CSS system was built around IBM Plex Mono, a token-based variable system, and a set of semantic aliases that made the dark/light toggle a one-line swap.

The design decisions were intentional. IBM Plex Mono is legible at small sizes, signals that this is a technical portfolio without requiring any decorative work, and renders consistently across operating systems without webfont loading issues at the sizes used. The token system — --bg, --surface, --border, --text, --text-dim, --hi — means the entire theme is six variables. Light mode is a six-line override block.

The GitHub contribution chart

The contribution chart on the about page is an SVG fetched directly from GitHub's API. GitHub returns a dark SVG. To make it render correctly in both light and dark mode without maintaining two versions, a CSS filter chain handles the inversion:

/* dark mode: SVG is already dark — enhance contrast */
filter: grayscale(1) contrast(1.2);

/* light mode: invert first, then enhance */
filter: grayscale(1) invert(1) contrast(1.2);

The grayscale(1) strips any residual color before the invert, which prevents hue shifts on colored contribution cells. The order matters — invert before grayscale produces different results than grayscale before invert.

Why the static approach held up

Static HTML with a shared stylesheet is fast, deployable anywhere, requires no build step, and has zero runtime dependencies. For a portfolio site with no authentication, no user state, and no dynamic data, it is the correct tool. The CSS system transferred directly to the migration — no rewrite required, just a different delivery mechanism.

Part 2 — Why Migrate

Four concrete problems drove the decision to migrate, not a preference for complexity:

Copy-paste navigation. Nine HTML files each contain the same nav block. Every link addition or active-state change requires editing nine files. This is the clearest possible signal that a component abstraction is needed.

The Vault. One section of this site should be password-gated. Static HTML has no mechanism for server-side authentication. Any client-side approach — hiding content with JavaScript, obscuring URLs — is not access control, it is theater. A Next.js API route can validate a password against an environment variable and return a token. That is real gating.

Mobile responsiveness. Several tables require horizontal scrolling on small screens. This is manageable with CSS, but a component-based approach makes it easier to swap in a different layout for mobile without duplicating markup.

Unity WebGL. The Chiaroscuro math visualization project will eventually embed a Unity WebGL build. Unity WebGL has a specific lifecycle — it requires a container element, calls into a JavaScript loader, and needs cleanup on unmount. A React component handles this cleanly. A static HTML page with an inline script is harder to maintain and test.

Part 3 — The Stack

The migration target is Next.js / React / TypeScript, built with Vite for local development. These are four separate things that are often conflated. They solve different problems at different layers.

React — what it actually solves

React is a UI library. It does not handle routing, data fetching, server rendering, or deployment. What it does: it lets you define a piece of UI as a function that takes props and returns markup, manages state changes without manually updating the DOM, and re-renders only what changed when state updates. The Nav component is the clearest example — define it once, import it everywhere, change it in one place.

Next.js — what it adds

Next.js is a framework built on React. It adds file-based routing (a file at pages/books.tsx becomes the /books route), server-side rendering and static generation, API routes (files in pages/api/ become serverless functions), and image optimization. The Vault password gate lives in a Next.js API route. The static pages become statically generated at build time — same performance as the HTML files, with the component model on top.

TypeScript — the compiler as a collaborator

TypeScript adds optional static types to JavaScript. The compiler catches a class of bugs before the code runs — wrong prop types, missing properties on an object, a function called with the wrong number of arguments. The cost is annotation overhead. The benefit is that refactoring is safer and the editor can provide accurate autocomplete.

The before/after is clearest with a component interface:

// Without TypeScript — silent failure
function BookRow(props) {
  return <tr><td>{props.titel}</td></tr>; // typo: titel
}

// With TypeScript — compiler error at the typo
interface BookRowProps {
  title: string;
  author: string;
  priority: 'high' | 'med' | 'low';
}

function BookRow({ title, author, priority }: BookRowProps) {
  return <tr><td>{title}</td><td>{author}</td></tr>;
}

The interface is a contract. The compiler enforces it. Every call site that passes a misspelled prop or the wrong type is flagged before the browser ever sees the code.

Vite — the build tool

Vite handles the local development server and the production build. It provides hot module replacement — changing a component file updates the browser instantly without a full reload. It is largely invisible during development, which is the goal of a build tool. It is faster than webpack for local dev because it uses native ES modules during development and only bundles for production.

Why not Tailwind CSS

The existing CSS system — custom properties, semantic aliases, the component classes — transfers directly to the React migration. There is no functional problem that Tailwind solves here that is not already solved. Rewriting every class declaration into utility strings would mean learning a new syntax to produce the same output. The CSS system is the asset, not a liability.

Part 4 — Routing

In the static site, routing is the filesystem. books.html is served at /books.html. Next.js uses the same mental model with a cleaner URL: a file at pages/books.tsx is served at /books. The difference is that the file is a React component, not raw HTML.

The alternative is React Router — a library that handles routing client-side, where the URL changes without a server request and React swaps the component in. Next.js makes the choice for you: file-based routing by default, with the option to add dynamic routes via bracket syntax (pages/log/[slug].tsx matches /log/any-post-title). For this site, Next.js file-based routing is the right fit — the content structure maps directly to the file structure.

Part 5 — Testing

Three layers of tests cover the migrated site:

Vitest for unit tests on pure utility functions — slug generation, date formatting, any function that takes an input and returns an output with no side effects. Fast, runs in Node, no browser required.

Supertest for integration tests on the Next.js API routes — the Vault password check, any route that reads environment variables or interacts with external services. Tests the HTTP interface directly.

Playwright for end-to-end tests on critical paths — the Vault gate, nav links, mobile layout. Runs a real browser, slower, reserved for flows where the full browser environment matters.

A Vitest unit test for a utility function:

import { describe, it, expect } from 'vitest';
import { slugify } from '../lib/slugify';

describe('slugify', () => {
  it('lowercases and replaces spaces with hyphens', () => {
    expect(slugify('Building This Site')).toBe('building-this-site');
  });
  it('strips special characters', () => {
    expect(slugify('Week 13 — AI & Agentic Workflows')).toBe('week-13-ai-agentic-workflows');
  });
});

Part 6 — Deployment

The static site deploys to GitHub Pages — push to main, the files are served. The Next.js migration deploys to Vercel. The workflow is the same: push to GitHub, Vercel detects the push, runs the build, and deploys. The difference is that Vercel understands Next.js natively — API routes become serverless functions, environment variables are managed through the Vercel dashboard, and preview deployments are created automatically for every pull request branch.

The Vault password gate requires an environment variable (VAULT_PASSWORD) that is never committed to the repository. On Vercel, environment variables are set in the project settings and injected at build time and runtime. The API route reads process.env.VAULT_PASSWORD — it is never exposed to the client bundle.

This is the concrete reason static HTML cannot do what the Next.js migration does: server-side environment variable access requires a server. Vercel's serverless functions are that server.

Part 7 — What's Next

The migration is in progress as of the calendar Phase 1–2 window. The CSS system is complete and in production. The HTML files are the source of truth until the React components replace them page by page. The order: Nav component first (unblocks everything), then page-by-page migration starting with the simplest pages, then the Vault gate once the API route infrastructure is in place.

A site-3 build log will follow after the migration completes — covering what the component extraction revealed about the existing markup, what TypeScript caught during the migration, and how long each phase actually took versus the estimate.
