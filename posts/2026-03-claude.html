Fixing Broken Rows with Claude

March 2026 · HTML · CSS · debugging · workflow

The calendar page on this site has 30 expandable week rows across three phases. Click a row header and the domain grid drops down — five columns covering CS fundamentals, systems depth, language and tooling, commerce and backend, and teaching. It's the most structurally complex page on the site. It was also completely broken.

Clicking certain rows would cause other rows to resize. The month label column — a thin vertical strip with rotated text — would shift width unpredictably. Some rows expanded cleanly. Others caused the entire table geometry to lurch. I knew enough about CSS and HTML to understand that something about the table structure was wrong, but I had 800 lines of hand-written markup and no clear starting point for the audit.

I gave Claude this prompt:

The prompt, polished:

The calendar page accordion rows are behaving inconsistently on expand — some rows resize correctly while others cause surrounding rows and the month label column to shift width. The content should not change at all. Please audit the HTML structure and CSS, identify what's causing the instability, and fix it. Revise the stylesheet if necessary.

What I actually typed was closer to: "can you fix the calendar page? the rows keep breaking... revise the css if you need to... i'm not sure what's going wrong. don't change any of the content - the rows just resize randomly when you click on some of them"

Both prompts got the same result. That's the part worth noting.

What was wrong

The calendar was built using HTML <table> elements for the month blocks, with rowspan on the month label cell and display: table-row toggled by JavaScript to show and hide each week's content. The problem is that display: table-row is controlled by the browser's table layout engine, which recalculates every column width in the table every time a row appears or disappears. And because some week header rows had colspan="2" and others didn't, the engine was receiving contradictory column count signals on every toggle. The table geometry lurched to reconcile them.

The fix was to stop using a table for the accordion structure entirely. Each month became a flex container — a fixed-width label column on the left, a weeks column on the right. Each week became a self-contained div. The body panel toggles between display: none and display: block, which has no effect on any surrounding geometry. One row expanding cannot affect another row's dimensions. The month label column doesn't move because it isn't part of a table that's being recalculated.

Claude audited the markup, identified the colspan inconsistency as the root cause, explained why display: table-row toggling creates this class of problem, and rebuilt the structure — all 800 lines of it — without changing a word of content.

What this actually saved

I know how CSS and HTML interact. I've written enough of both to understand the box model, the table layout algorithm, and why flexbox is generally the right answer for anything that needs to expand and collapse. That knowledge wasn't the bottleneck. The bottleneck was time: finding the exact lines causing the issue across 800 lines of markup I wrote myself, weeks ago, while also tracking the rowspan arithmetic across four separate month tables and three phases.

A manual audit of that problem would have taken the better part of an afternoon — reading the markup, forming a hypothesis about the colspan inconsistency, testing the fix on one month, propagating the structural change across the rest, updating the CSS to remove the old table rules and add the new flex rules, verifying the toggle behavior, and checking the mobile breakpoints. The fix Claude produced took about four minutes and required one review pass on my end to confirm nothing had changed in the content.

That gap — between knowing what the problem is in principle and actually resolving it across a large hand-written file — is where AI tooling pays off most clearly. Not as a replacement for understanding the system, but as a force multiplier for applying that understanding efficiently.

The static site problem

This post exists partly because the calendar fix was a turning point in how I was thinking about the site. Before it, I was close to stopping updates to the static HTML version altogether. Not because the content wasn't worth maintaining, but because the maintenance loop was becoming untenable.

The navbar was the clearest symptom. The hamburger menu worked on index, books, and calendar — but only because those three pages happened to have the right JavaScript. The other five pages had a static nav with no mobile drawer, no toggle button, no working menu on small screens. Fixing that meant opening eight files, finding each nav block, pasting in the corrected markup, verifying the active link was right for each page, adding the script tag, and saving. Mechanical work. The kind that accumulates until you stop doing it.

Without a component system — without React, without even a templating layer — every change to a shared element means touching every file that contains it. That's the core argument for migrating to Next.js, and the migration is already on the calendar. But the static site is what recruiters and students see now, and it needed to work now.

Claude fixed the navbar in the same session as the calendar: one snippet per page, pre-labeled, with the correct active class already set and the drawer markup included. Fifteen minutes of copy-paste instead of an afternoon of careful manual editing. The site works on mobile. The calendar rows expand cleanly. The nav is consistent across all eight pages.

None of that required delegating understanding. It required delegating the mechanical application of understanding I already had — and that's a distinction worth being precise about, because the alternative framing ("AI wrote my website") is both wrong and uninteresting.

On prompting precisely

The polished version of my prompt at the top of this post is not what I actually typed. What I typed was a rough description of symptoms with no technical detail. Claude audited the markup, found the root cause, explained it, and fixed it — from the rough description.

That's a reasonably good argument for not over-engineering your prompts before you know what the problem is. Describe the behavior, not your theory about the cause. Let the tool do the audit. Verify the explanation before accepting the fix. That sequence — describe symptom, receive diagnosis, verify reasoning, accept output — is faster and more reliable than spending time constructing a technically precise prompt for a problem you haven't fully characterized yet.

The calendar page works. The next post will be about something more interesting than broken tables.

The calendar is being migrated to Next.js in Phase 2 of the plan — at that point the accordion becomes a proper React component and this class of structural problem goes away permanently. The static fix is a bridge, not a destination.
