Coffee Passphrases — Secure by Design

About · Mar 4, 2026

· Python · PBKDF2 · cryptography

This site has two gated sections: a student-facing homepage and a vault for close friends. Both need passwords — ones I can share over text or say out loud without either of us reaching for a pen. This is the script I wrote to generate them, and an explanation of why each decision was made the way it was.

The wordlist is drawn entirely from James Hoffmann's The World Atlas of Coffee — processing methods, varietals, tasting descriptors, brew terms. The output looks like doppio-clean-demucilaged-caturra. Speakable. Typeable on a phone. Memorable enough to survive switching windows. And because the vocabulary is specific to a domain most wordlist attacks have never heard of, it holds up better than it might look.

The passphrase itself is simple. What happens to it before storage is not.

Why a passphrase over a character string

The first version of this script generated passwords like #cjK/>l}6<^rC]!@]U!/. Technically strong. Completely unusable in practice for this context.

These passwords get shared directly with students and recruiters — over text, in person, sometimes read aloud. A character string fails all three. A passphrase like doppio-clean-demucilaged-caturra survives a text message, is readable at a glance, and is memorable enough that someone can close the message and still get in a few minutes later without hunting for it.

The entropy tradeoff is acceptable here. Four words drawn from categorized pools of ~20–30 words each gives roughly 40–50 bits of entropy — less than the character string, more than enough for a low-stakes shared credential. The threat model is curious people who shouldn't have access, not adversarial attackers with GPU clusters running dictionary attacks.

The wordlist comes from Hoffmann's vocabulary specifically because it isn't in standard attack wordlists. Demucilaged and maragogipe do not appear in rockyou.txt.

Pattern: brew · descriptor · processing · varietal

Rather than pulling all four words from a flat list, the script draws one word from each of four categories so the result reads as a loose coffee narrative. doppio-clean-demucilaged-caturra tells a small story: a brew method, a cup quality, a processing decision, a varietal. That narrative structure is part of what makes it memorable.

This is slightly lower entropy than drawing all four from a single large pool — you're sampling from smaller buckets — but the usability gain is worth it. A password your student actually pastes in is more secure than one they transcribe wrong and ask you to resend.

The problem with storing a raw password

The naive approach is to put a password in a .env file and compare it directly against what the user submits. That works — until your environment variables leak, your repo gets exposed, or someone with access to your Vercel dashboard screenshots it. A raw password is immediately useful to anyone who sees it.

A hash changes that. Instead of storing mypassword, you store the output of running it through a one-way function. When a user submits a password, you hash what they sent and compare the two hashes. The original password never needs to be stored anywhere.

Why salt matters

Hashing alone has a known weakness: identical passwords produce identical hashes. If someone builds a table of common passwords and their SHA-256 outputs ahead of time (a rainbow table), they can reverse your hash instantly by lookup.

A salt is a random value generated fresh for each password and combined with it before hashing. Two users with the same password get completely different hashes because their salts differ. It makes precomputed attacks useless — the attacker would need to rebuild their entire table for every possible salt, which is computationally infeasible.

The salt doesn't need to be secret. It's stored alongside the hash. Its only job is to be unique and random.

PBKDF2 — deliberately slow by design

A regular hash like SHA-256 is fast — that's usually what you want from a hash function. For password storage it's a liability. A modern GPU can compute billions of SHA-256 hashes per second, which makes brute-force attacks practical.

PBKDF2 (Password-Based Key Derivation Function 2) runs the hash function repeatedly — 600,000 times in this script, the current NIST recommendation. That turns a microsecond operation into something that takes real time, which scales poorly for an attacker guessing millions of candidates but is imperceptible to a legitimate user logging in once.

The iteration count is stored in the output string alongside the salt and hash. That means if NIST raises the recommendation in two years, existing hashes don't break — you just use the stored count when verifying old hashes and the new count for new ones.

The output format

Each generated hash is stored as a single string in this format:

pbkdf2$600000$<salt_hex>$<hash_hex>

Everything needed to verify a future password submission is in that one string: the algorithm family, iteration count, salt, and derived key. No separate columns, no separate lookups. This is the same convention used by Django's password hasher and several other production systems.

Timing attacks — one more thing

There's a subtle attack worth knowing about: if your password comparison returns faster when the first character is wrong than when the last character is wrong, an attacker can measure response times to learn how close their guess was character by character.

The fix is hmac.compare_digest() — a comparison function that takes the same amount of time regardless of where the strings differ. One line, but it closes a real attack surface. It's in Python's standard library.

The script

Two files: coffee_words.py holds the categorized wordlist. gen_passwords.py imports from it, generates the passphrases, hashes them, verifies them, and prints the output. No third-party dependencies — everything is in Python's standard library.

python gen_passwords.py

The raw passphrases print once to your terminal. The script does not write them to disk. Save them to a password manager immediately — Bitwarden is free, open source, and audited. Then paste the hash strings into .env.local and your Vercel environment variables dashboard.

Note on the Next.js side: the API routes that receive password submissions use the same PBKDF2 logic reimplemented in Node's built-in crypto module — same algorithm, same iteration count, same salt format. The verification function also uses Node's crypto.timingSafeEqual() for the same reason Python uses hmac.compare_digest(). The two implementations are interoperable because PBKDF2-HMAC-SHA256 is a standard, not a library-specific format. The cultural reasoning behind the wordlist — why Vienna gets its own separate vocabulary — is in the companion post: The Room Where Ideas Happen.

In practice vs. at scale

This implementation is appropriate for what it is: a personal portfolio site with two static passwords protecting low-stakes access. Understanding why it would change at scale is the more useful exercise.

Password management — human credentials

Most working engineers split credential storage into two categories that don't overlap. A password manager (Bitwarden, 1Password) handles personal authentication — your GitHub login, Vercel account, email. A secrets manager (AWS Secrets Manager, HashiCorp Vault, Vercel environment variables) handles runtime credentials — API keys, database URLs, tokens your code needs. The distinction matters because the threat models differ: human credentials get phished, runtime secrets get leaked through source code or logs.

Bitwarden is worth knowing about specifically because it is fully open source (AGPL licensed), independently audited, and self-hostable. The encryption implementation is public and readable — which is the open source value proposition applied concretely to something most people take on faith.

Hardware keys

A YubiKey implements FIDO2/WebAuthn — it's a physical second factor that cryptographically signs a challenge that includes the requesting site's domain. It cannot be phished because it will not sign a challenge from a domain it hasn't seen before. Even with your password, an attacker cannot authenticate without the physical key.

The right sequencing before getting one: move all accounts into a password manager first, enable TOTP-based 2FA on everything that matters, then add a hardware key to your highest-value accounts (email, GitHub, Vercel). A YubiKey on an account with a reused weak password doesn't help. The password manager is the foundation.

Passphrases vs. generated passwords

For passwords you need to memorize, a passphrase — four or five random unrelated words — is stronger than a random character string of comparable length. Length dominates entropy, and passphrases are longer in practice because memorability stops being a constraint. For passwords stored in a password manager and never typed, the generated approach this script uses is marginally better since you're not trading off memorability for anything.

The meaningful variable in both cases is uniqueness per site. Credential stuffing — taking a leaked username/password pair from one breach and trying it everywhere else — is responsible for far more account compromises than weak passwords are.

What changes for real user accounts

The conceptual pieces here — salt, slow hash, timing-safe comparison, httpOnly cookie — are the same ones production systems use. The architecture around them changes:

Algorithm: Argon2id instead of PBKDF2. Argon2id won the Password Hashing Competition in 2015 and is the current recommendation for new systems. The meaningful difference from PBKDF2 is that Argon2id is memory-hard — it requires a configurable amount of RAM per computation, which makes GPU-parallel brute force attacks expensive in a way that PBKDF2's CPU-only cost does not. In Node: the argon2 package. In Python: argon2-cffi.

Where hashing happens: In a real registration flow, the raw password is hashed in the same server process that receives the form submission and is never written anywhere — not to logs, not to a terminal, not to a temp file. The hash is what gets stored. This script prints the raw password to the terminal as a one-time setup convenience; that tradeoff is only acceptable because these are site-level shared secrets, not user credentials.

Account lockout: After a configurable number of failed attempts, the account gets locked and requires email verification to unlock. This makes the timing-safe comparison largely moot in practice — an attacker can't make enough guesses to exploit timing differences if they're locked out after five attempts.

Session tokens: The vault cookie in this implementation is static — the same value for the entire 24-hour session. Production sessions rotate the token on each request or on a short TTL, so a stolen token has a narrow usable window. The cookie also gets a new value on each login, which invalidates any previously captured tokens.

Password reset: Through a signed, time-limited token sent to a verified email address — not a shared secret. The token is single-use and expires after 15–30 minutes. There is no "what is your mother's maiden name" fallback.

The point of doing this manually here is to understand what the libraries are doing. NextAuth, Auth.js, and Clerk implement all of the above — once the conceptual model is clear, using them is a responsible choice rather than a black box.
